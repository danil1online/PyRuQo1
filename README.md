# o1_gigachat-20b-a3b_lora_npi

Разработка основана на [Russian o1 / GigaChat 20B-A3B Instruct](https://huggingface.co/evilfreelancer/o1_gigachat-20b-a3b_lora), где обобщенно показано, как создан LoRA-адаптер для модели GigaChat-20B-A3B (и видимо для o1) на датасете, который, видимо, сейчас уже является [Dataset_of_Russian_thinking](https://huggingface.co/datasets/Egor-3926/Dataset_of_Russian_thinking). 

Программа, повторяющая данное обучение предствалена в [репозитории](scripts/basic_example.py). Скрипт оптимизирован под жесткий лимит в 24 ГБ VRAM (RTX 3090): в нём активированы 4-битное квантование базы, сохранение контрольных точек градиентов (gradient_checkpointing) и страничный оптимизатор, который убережет карту от ошибки Out of Memory.

Порядок запуска "базового скрипта":
```bash
git clone http://195.133.13.56/danil1online/o1_gigachat-20b-a3b_lora_npi.git
cd o1_gigachat-20b-a3b_lora_npi

source env/bin/activate
pip install -r requirements.txt
python scripts/basic_example.py
```

**Если скрипт падает с ошибкой Out of Memory**. Если в процессе обучения видеокарта всё же переполняется, уменьшите max_seq_length до 1024 в блоке SFTTrainer. Длина контекста сильнее всего влияет на потребление VRAM при обучении.

**Как адаптировать под университетские публикации**. Когда будет сформирован собственный датасет из статей (например, с помощью o1), структура JSON файла должна соответствовать функции [formatting_prompts_func](scripts/basic_example.py#L34). # Создание датасета

В датасете [Dataset_of_Russian_thinking](https://huggingface.co/datasets/Egor-3926/Dataset_of_Russian_thinking) каждая строка состоит из трех текстовых полей: system, prompt (запрос пользователя) и response (ответ системы, содержащий теги <Thought>...</Thought> и сам ответ).

Чтобы превратить n-летний архив научных публикаций университета в такой датасет, нужно решить главную задачу: научить модель рассуждать на основе ваших текстов, а не просто зазубривать их.

Ниже представлено пошаговое руководство и готовый автоматизированный скрипт для генерации такого датасета.

## Архитектура генерации датасета

Прямой перевод научных статей в формат «Промпт — Ответ» строится по принципу генерации синтетических данных (Synthetic Data Generation). В качестве «генератора» можно использовать o1_gigachat (или любую другую сильную reasoning-модель), запуская её на CPU.

Процесс разбивается на 4 этапа:

1. Чанкинг (Нарезка): Статья бьется на смысловые фрагменты (например, по 3000–4000 символов).
2. Промптинг генератора: Мы просим модель o1 прочитать кусок статьи и придумать сложный, требующий размышлений вопрос к нему.
3. Генерация цепочки рассуждений: Мы заставляем o1 подробно расписать свои мысли (для тега <Thought>), а затем сформулировать итоговый ответ.
4. Экспорт: Данные упаковываются в JSON, полностью готовый для SFTTrainer.

### Обработка сырых данных. Пример: "куча" PDF файлов

Для работы с «кучей PDF» на Ubuntu 22.04 лучше всего использовать библиотеку [PyMuPDF (fitz)](https://github.com/pymupdf/pymupdf). Она работает в разы быстрее, чем старый [PyPDF2](https://pypi.org/project/PyPDF2/), корректно склеивает переносы строк и извлекает текст из многоколоночных академических версток без превращения их в кашу. 

В репозитории [представлен Python-модуль](scripts/pdf_parser.py), который автоматически обойдет все папки с вашими распознанными PDF-файлами, очистит текст от специфического «мусора» (колонтитулы, списки литературы, метаданные) и наречет его на аккуратные смысловые чанки.

Данный скрипт "парсинга и чанкинга" [pdf_parser.py](scripts/pdf_parser.py) берет PDF-файлы, склеивает разорванные из-за переносов слова (например, нано- \n технологии превращает в нанотехнологии) и режет текст с «нахлестом» (overlap), чтобы логическая мысль на стыке страниц не разрывалась.

**Примечание по «сканам»** Если среди n-летних архивов есть сканированные копии документов (внутри PDF лежат просто картинки без текстового слоя), скрипт pymupdf пропустит их (выдаст размер текста < 200 символов). Если таких файлов много, их придется предварительно пропустить через систему оптического распознавания текста (OCR). На Ubuntu это проще всего сделать утилитой `ocrmypdf` или `tesseract-ocr-rus`, которая перезапишет PDF, добавив в них текстовый слой поверх картинок:

```bash
sudo apt update && sudo apt install ocrmypdf tesseract-ocr-rus -y
ocrmypdf input_scanned.pdf output_searchable.pdf -l rus
```

### Пояснение как скорректировать под это скрипт обучения

Поскольку мы уже на этапе сборки датасета зашили теги <Thought> и <output> внутрь поля response (как это сделано в оригинале), [функция форматирования в базовом скрипте обучения](scripts/basic_example.py#L34) станет еще проще. Не нужно будет вручную собирать строки из кусочков, достаточно будет просто прочитать готовые поля:
```python
# Новая функция форматирования для SFTTrainer под ваш готовый JSON:
def formatting_prompts_func(examples):
    output_texts = []
    for i in range(len(examples['prompt'])):
        text = (
            f"Система: {examples['system'][i]}\n\n"
            f"Пользователь: {examples['prompt'][i]}\n\n"
            f"Система: {examples['response'][i]}"
        )
        output_texts.append(text)
    return output_texts
```

### Стратегический совет по типам вопросов

Чтобы университетская модель получилась разносторонней, настройте промпт-генератор так, чтобы на каждые 10 чанков статьи он генерировал разные типы заданий:
- 30% — Вопросы "Почему/Как" (проверка глубокой логики и причинно-следственных связей).
- 30% — Саммаризация (Промпт: "Сократи этот научный текст до 3 предложений", где в <Thought> модель анализирует, какие факты главные, а какие второстепенные).
- 20% — Поиск противоречий или ограничений метода (Промпт: "Какие слабые места есть в методологии авторов текста?").
- 20% — Извлечение терминов и сущностей (Формирование базы знаний).

### [Итоговый скрипт сборки датасета](scripts/build_dataset_via_api.py)

Предполагается, что уже выполнен инференс модели для генерации (o1) через REST API сервера llama.cpp (совместимого со стандартным API OpenAI). Это позволяет полностью автоматизировать и распараллелить процесс генерации данных. Поскольку инференс o1-модели вынесен на сетевой адрес, скрипт сборки датасета не будет тратить ресурсы локального процессора на генерацию, а займется исключительно парсингом PDF и логистикой данных. В [файле](scripts/build_dataset_via_api.py) представлен рабочий скрипт, который объединяет в себе:

- Рекурсивный обход папок и парсинг PDF (с обработкой сканов через OCR, если текстовый слой пуст).
- Сетевые асинхронные запросы к вашей модели по адресу http://195.133.13.56:8079/v1/chat/completions.
- Сборку структуры system / prompt / response с тегами <Thought> для точного повторения логики оригинального датасета.

Эта схема реализует: 
- Защиту по JSON-mode: Параметр "response_format": {"type": "json_object"} нативно поддерживается встроенным сервером llama.cpp. Он гарантирует, что модель не выдаст текстовый мусор в начале ответа, и функция json.loads никогда не сломается.
- Автосохранение: Скрипт записывает файл каждые 10 чанков. Если у вас моргнет интернет или удаленный сервер перезагрузится, вы не потеряете сгенерированные за несколько часов данные. 

Если есть огромный архив публикаций (например, более 10 000 чанков), генерация в один поток через цикл может занять много времени. Можно переписать блок генерации на многопоточность (библиотека concurrent.futures), но это целесообразно, если llama.cpp сервер мощный. 

Предположим вместо этого наличие второго независимого сервера. Серверы работают автономно друг от друга, их мощности можно объединить без риска перегрузить какой-то один из них. Для этого применяется классический паттерн балансировки нагрузки (Load Balancing) на стороне клиента. Ваш локальный ПК будет распределять чанки из очереди по двум серверам параллельно. Это сократит общее время сборки датасета ровно в 2 раза.В файле [build_dataset_multi_server.py](scripts/build_dataset_multi_server.py) представлена обновленная, многопоточная версия модуля генерации. Она использует пул потоков (ThreadPoolExecutor), распределяет задачи по пулу доступных URL и автоматически подменяет упавший или зависший сервер, если с ним пропадет связь.

Главные преимущества такой архитектуры:
- Строго по одному запросу на сервер: Структура queue.Queue() и ограничение max_workers=len(SERVERS_POOL) гарантируют, что на каждый сервер в один момент времени идет ровно один запрос. Слабые сервера будут защищены от переполнения VRAM.
- Асинхронность: Если первый сервер считает короткий чанк и освобождается за 30 секунд, а второй завис на сложном графике на 2 минуты, первый сервер не будет ждать второго. Он сразу заберет из очереди следующий чанк.
- Отказоустойчивость: Если один из серверов физически отключится в процессе работы, скрипт продолжит генерировать данные через оставшийся рабочий сервер (хоть скорость и упадет до однопоточной).

Если в будущем появится третий или четвертый подобный сервер, достаточно будет просто дописать их IP-адреса в массив SERVERS_POOL — скрипт автоматически масштабирует потоки под новое количество узлов.

## [Скрипт обучения на собственном датасете](scripts/train_qlora.py) 
сохранит только маленькие веса адаптера (~100-200 МБ). Чтобы использовать модель полноценно (например, конвертировать её в GGUF для инференса), потребуется запустить [скрипт слияния (Merge)](scripts/merge_lora_example.py) базовой модели GigaChat-20B и сохраненного адаптера. 

Cлияние (Merge) нужно делать строго до финального квантования в GGUF (если такое планируется). Инструменты конвертации (такие как llama.cpp) не умеют напрямую склеивать 4-битные квантованные файлы GGUF с отдельными LoRA-адаптерами.

Процесс устроен так: 
- базовая модель [GigaChat-20B-A3B-instruct-v1.5-bf16](https://huggingface.co/ai-sage/GigaChat-20B-A3B-instruct-v1.5-bf16) загружается в исходной точности (BF16), 
- к модели применяется адаптер, 
- веса суммируются, 
- результат сохраняется как полноценная новая модель (BF16), 
- полноценная новая модель затем квантуется в GGUF.

Поскольку базовая модель GigaChat-20B-A3B в формате FP16 весит около 40 ГБ, она не поместится целиком в видеокарту RTX 3090 (24 ГБ) во время слияния. Однако если есть **256 ГБ** системной RAM и мощный процессор, то следующий скрипт задействует оперативную память (CPU) для выполнения этой операции, вообще не перегружая видеокарту.

```bash
python scripts/merge_lora_example.py
```

Предположим, такого ПК нет. Есть **64 ГБ** оперативной памяти. Это абсолютный технологический минимум, но его хватит для слияния модели GigaChat-20B-A3B (размер в FP16 составляет около 39–40 ГБ). Однако, если запустить процесс «в лоб», операционная система задействует файл подкачки (Swap), из-за чего слияние будет идти очень медленно или завершится аварийно (ошибкой Out of Memory в ОС). Чтобы слияние гарантированно прошло успешно на 64 ГБ ОЗУ, скрипт необходимо оптимизировать. **Главный секрет**: нельзя загружать модель в память одним куском. Hugging Face умеет читать и обрабатывать модель послойно прямо с диска. 

**Оптимизированный скрипт** слияния для 64 ГБ RAM [merge_low_ram.py](scripts/merge_low_ram_example.py). 

Этот вариант использует параметр low_cpu_mem_usage=True и пошаговую склейку слоев, снижая пиковое потребление ОЗУ до ~42–45 ГБ.

Важные правила подготовки ПК перед запуском:
- Закройте все приложения, браузеры (особенно Chrome), мессенджеры и среды разработки (IDE вроде PyCharm). Нужно освободить максимум из 64 ГБ.
- Настройте файл подкачки (обязательно). Если используется Linux убедитесь, что размер Swap составляет не менее 30–40 ГБ (желательно на быстром NVMe SSD). Если Windows, зайдите в Параметры системы -> Быстродействие -> Виртуальная память и жестко задайте размер файла подкачки на диске C: (или там, где стоит SSD) в диапазоне от 32000 до 48000 МБ.
- Запаситесь терпением. Из-за процессора i5-9400F и работы на грани лимита оперативной памяти процесс сохранения модели на диск может занять от 15 до 30 минут. Это нормально, главное — не прерывать скрипт. 

Процессы обучения и слияния физически разделены во времени и происходят по разным математическим принципам. 
1. LoRA-адаптер — это отдельный файл весов. При обучении методом QLoRA базовая модель действительно сжимается до 4 бит, но сами веса LoRA (матрицы A и B) обучаются и сохраняются в исходной высокой точности (FP16/BF16). На диск записывается только этот маленький адаптер (~100-200 МБ), а 4-битная база просто выбрасывается из памяти.
2. Как происходит слияние (Merge). В скрипте слияния вы загружаете оригинальную, чистую базовую модель с Hugging Face сразу в формате FP16 (без всякого квантования). Затем поверх неё накатывается ваш сохраненный LoRA-адаптер (который тоже в FP16). Поскольку их типы данных совпадают, они без проблем математически складываются (W_new = W_base + A * B).

**Важное предостережение для GigaChat-20B-A3B**

Архитектура моделей семейства GigaChat-20B имеет одну важную специфику, о которой необходимо знать до запуска скрипта слияния. В отличие от стандартных моделей LLaMA или Mistral, базовый скрипт merge_and_unload() из стандартной библиотеки peft может выдать ошибку или отработать некорректно, если модель использует проприетарный код архитектуры (trust_remote_code=True), как в случае со Сбером. Если при выполнении стандартного скрипта слияния вы столкнетесь с ошибкой в строке model.merge_and_unload(), используйте проверенный альтернативный путь, который гарантированно работает для GigaChat:

```python
# АЛЬТЕРНАТИВНЫЙ ВАРИАНТ (если стандартный merge_and_unload выдает ошибку):
# Вместо merged_model = model.merge_and_unload() используем базовые методы:

print("Применение адаптера к базовой модели...")
# Этот метод переносит веса LoRA внутрь основных слоев, но оставляет структуру PEFT
model.base_model.merge_adapter() 

# Извлекаем чистую внутреннюю модель, которая теперь содержит объединенные веса
merged_model = model.unload() 
```
Этот обходной путь делает ровно то же самое: физически объединяет матрицы в геометрии FP16, подготавливая модель к корректной конвертации в GGUF через llama.cpp.

### Конвертация в GGUF

Как только скрипт отработает, у вас появится папка ./merged_o1_gigachat_university с файлами safetensors. Чтобы превратить её в один файл .gguf для запуска на CPU+GPU через llama.cpp или Ollama, выполните в терминале следующие шаги:

1. Клонируйте официальный репозиторий llama.cpp и установите его зависимости:
```bash
https://github.com/ggml-org/llama.cpp.git
cd llama.cpp
pip install -r requirements.txt
```

2. Сконвертируйте модель из формата Hugging Face в формат GGUF (изначально в максимальном качестве FP16):
```bash
python convert_hf_to_gguf.py ../merged_o1_gigachat_university --outfile ../university_model_f16.gguf
```

3. Сквантуйте (сожмите) полученный файл до оптимального для вашей RTX 3090 размера (рекомендуется метод Q4_K_M, он весит около 13-14 ГБ и отлично встанет в 24 ГБ VRAM, сохранив интеллект модели):
```bash
./llama-quantize ../university_model_f16.gguf ../university_model_Q4_K_M.gguf Q4_K_M
```

Финальный файл university_model_Q4_K_M.gguf будет полностью готов к работе.

## Итоговый порядок запуска, а также минимальные технические требования

Идеальная база -- Ubuntu 22.04. На этой ОС управлять памятью и автоматизировать процессы гораздо проще, чем на Windows.

Ниже пошаговое руководство, как подготовить систему, запустить скрипт слияния и затем конвертировать модель в GGUF, не вызывая критической ошибки Out of Memory (когда процесс убивается системой через OOM Killer).

 1.2. Установка Python-зависимостей репозитория

 1.3. Подготовка данных и запуск сборки датасета

 1.3.1. Создайте в корне каталога или укажите путь к папке с вашими PDF-документами (например, ./university_pdfs). Положите туда файлы научных публикаций за последние 10 лет.

 ```bash
 python3 scripts/build_dataset_multi_server.py
 ```
 Скрипт начнет обход папок, автоматически применит OCR к «слепым» сканам, нарежет тексты на чанки по 3500 символов и параллельно отправит запросы на сервера http://195.133.13.56:8079 и http://195.63.145.3:8078.
 1.3.3. На выходе в корне появится файл university_thinking_dataset.json, содержащий пары «Вопрос — Рассуждение <Thought> — Ответ».

2. QLoRA-обучение модели (Fine-Tuning)

Этот этап выполняется строго на ПК с видеокартой RTX 3090 (24 ГБ VRAM).
Скрипт автоматически скачает базовую модель GigaChat-20B-A3B-instruct-v1.5, сквантует её в 4 бита для экономии памяти видеокарты, подключит ваш созданный датасет и начнет тренировку.
```bash
python3 scripts/train_qlora.py
```
После завершения обучения в каталоге проекта будет создана папка ./o1_gigachat_university_lora. В ней будут лежать обученные веса адаптера (матрицы низкого ранга) объемом около 100–200 МБ.

3. Слияние весов (Merge) и создание GGUF

Процесс выполняется на ПК с 64 ГБ RAM и Ubuntu 22.04.

 3.1. Настройка SWAP (файла подкачки)

 Базовая модель весит ~40 ГБ в формате FP16 (p. 2). Чтобы предотвратить аварийное завершение процесса менеджером памяти Linux (OOM Killer), создайте временный swap-файл на 40 ГБ на быстром NVMe SSD (п. 1):
 ```bash
 # Отключаем текущий swap, если он активен
 sudo swapoff -a

 # Создаем файл подкачки размером 40 ГБ
 sudo dd if=/dev/zero of=/swapfile bs=1G count=40 status=progress

 # Устанавливаем права доступа и форматируем
 sudo chmod 600 /swapfile
 sudo mkswap /swapfile

 # Включаем новый swap и проверяем
 sudo swapon /swapfile
 free -h
 ```

 3.2. Вызов скрипта низкопотребляемого слияния весов

 3.2.1. Измените параметр агрессивности использования подкачки, чтобы ОС заранее разгружала RAM (п. 1):
 ```bash
 sudo sysctl vm.swappiness=80
 ```
 3.2.2. Запустите оптимизированный скрипт слияния из репозитория (п. 1):
 ```bash
 python3 scripts/merge_low_ram_example.py
 ```
 Скрипт загружает базовую модель послойно (не всю сразу), подтягивает веса обученного адаптера из ./o1_gigachat_university_lora, математически складывает их и сохраняет полноценную FP16-модель частями по 2 ГБ в папку ./merged_o1_gigachat_university.

4. Конвертация и Квантование в формат GGUF

 4.1. Подготовка компилятора и сборка llama.cpp

 Для компиляции инструментов конвертации соберите llama.cpp с поддержкой видеокарты для максимального ускорения (п. 2):
 ```bash
 sudo apt install build-essential cmake git python3-pip -y
 git clone https://github.com/ggml-org/llama.cpp.git
 cd llama.cpp

 # Сборка проекта с поддержкой CUDA под вашу RTX 3090
 cmake -B build -DGGML_CUDA=ON
 cmake --build build --config Release -j$(nproc)

 # Установка зависимостей конвертера
 pip3 install -r requirements.txt
 ```

 4.2. Конвертация Hugging Face в GGUF (FP16)

 Переведите сохраненную на этапе слияния модель в монолитный файл формата GGUF (п. 2):

 ```bash
 python3 convert_hf_to_gguf.py ../merged_o1_gigachat_university --outfile ../university_model_bf16.gguf
 ```
 Будет создан файл university_model_f16.gguf размером ~40 ГБ.

 4.3. Финальное квантование (Сжатие модели)

 Сквантуйте модель в формат Q4_K_M (оптимальный баланс размера и интеллекта для 24 ГБ видеопамяти):

 ```bash
 ./build/bin/llama-quantize ../university_model_f16.gguf ../university_model_Q4_K_M.gguf Q4_K_M
 ```
 или
 ```bash
 ./build/bin/llama-quantize ../university_model_bf16.gguf ../university_model_Q8_0.gguf Q8_0
 ```

 Итог: В корневой директории появится готовый к локальному инференсу файл university_model_Q4_K_M.gguf размером около 13–14 ГБ.

 4.4. Очистка временных ресурсов

 После проверки работоспособности файла удалите временный swap-файл и освободите место на SSD.
 ```bash
 sudo swapoff /swapfile
 sudo rm /swapfile
 ```
